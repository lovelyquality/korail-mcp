## 1. 한 줄 요약

> 한국철도공사가 공공데이터포털에 개방한 341건의 데이터를 **품질·시의성 기준으로 선별(136건 사용 / 205건 제외)** 한 뒤, **데이터 성격별 MCP 서버 10개**와 **직원용·고객용 AI Agent 6종**으로 재구성하여, 검색·해석·의사결정 지원이 가능한 대화형 서비스로 전환한다.

---

## 2. 추진 배경

### 2.1 외부 환경
- 정부 차원의 **'소버린 AI(Sovereign AI)'** 전략 추진과 함께 공공기관 보유 데이터의 AI 활용 압력이 증가하고 있음.
- 챗GPT·Claude 등 범용 LLM이 KORAIL 내부 운영 맥락을 모른다는 점은 명백한 격차이며, 이를 메우는 도구가 **MCP(Model Context Protocol)** 기반 데이터 연결과 도메인 특화 Agent임.

### 2.2 내부 환경
- KORAIL은 공공데이터포털에 **341건의 데이터셋을 개방**하고 있으나, 다음과 같은 한계가 있음:
  - 다운로드 후 직접 가공해야 사용 가능한 **정적 파일이 96%** (REST API는 7건뿐)
  - **60%가 1년 이상 미갱신**되어 사용자가 시의성을 판단하기 어려움
  - 데이터 간 연결성 없이 개별 파일로 흩어져 있어 **통합 분석·검색 불가**

### 2.3 문제 인식 한 줄 요약

> "데이터는 있지만, 누구도 한 번에 검색하고 해석할 수 없는 상태."

## 5. 데이터 도메인 분류 (MCP 서버 단위)

사용 가능 136건을 데이터 성격별로 **10개 MCP 서버**로 재편한다.

> **MCP 서버란?** AI가 외부 데이터·도구를 호출하기 위한 표준 통로. 도메인별로 1개씩 만들면, Agent가 필요한 도메인만 골라 호출할 수 있어 유지보수가 쉬워진다.

| # | MCP 서버명 | 데이터 성격 | 주요 데이터 (예시) | 건수 | 서비스 대상 |
|---|---|---|---|---:|---|
| M1 | `mcp-train-ops` | 운행·시간표·예약 | 차세대예약발매 시리즈, 일반열차·전철 여객발착 정보, 지연열차 승차권 | 약 23 | 직원+고객 |
| M2 | `mcp-station-facility` | 역·시설·접근성 | 역 위치, 역사 내외부 시설현황, 승강설비, 환승현황, 역상세 | 약 15 | 직원+고객 |
| M3 | `mcp-stats-passenger` | 여객 수송·이용 통계 | 역별 승하차, 수도권 전철 수송실적, 광역철도 수송인원, KTX 구간별 통계 | 약 15 | 직원 |
| M4 | `mcp-rolling-stock` | 차량·장비·제원 | 동력차 형별제원, 화차·기계 보유현황, 차량정비 실적, 전동차구간 정보 | 약 11 | 직원 |
| M5 | `mcp-freight` | 화물·물류 운영 | 화물 수탁·적하, 컨테이너 규격, 화물정보 데이터, 물류시설 | 약 10 | 직원 |
| M6 | `mcp-network` | 노선·운영 인프라 | 철도운행거리(노선별), 노선정보, 역간최단거리, 임율, 표준적하시간 | 약 15 | 직원 |
| M7 | `mcp-codebook` | 코드·용어·규정 | 철도통계 용어정의, 공통코드, 역코드, 차량유형코드, 표준방송 분류코드 | 약 10 | 모든 Agent 공통 |
| M8 | `mcp-internal-svc` | 직원·내부 운영 | 직원숙사 예약·앱·공지, 사회공헌, 임대자산, 정보공개 부서·공지 | 약 15 | 직원 |
| M9 | `mcp-procurement` | 조달·물품·자재 | 조달 입찰공고·발주계획, 물품정보, 자재그룹코드, 관계법령 | 약 12 | 직원 |
| M10 | `mcp-voc-cs` | 고객·홍보·정보공개 | VOC 시스템 통계, 고객의소리 만족도, 사전정보공표, 표어 | 약 10 | 직원+고객 |

> **유의 사항.** M1~M10의 건수 합계가 136과 정확히 일치하지 않는 이유는, 일부 데이터가 두 도메인에 동시 적재되기 때문이다(예: '역코드'는 M2와 M7 양쪽).

## 6. AI Agent 그룹화 방안

### 6.1 Agent 페르소나 정의 (6종)

#### 🎫 고객용 Agent

**C1. 여행·예매 도우미 Agent**
- 사용자: 일반 고객 (모바일 코레일톡, 홈페이지)
- 기능: "강릉 가는 KTX 토요일 오전 표 알려줘", "수원역 첫차 시간", "ITX-새마을 운임"
- 호출 MCP: M1, M2, M7

**C2. 접근성·역 안내 Agent**
- 사용자: 휠체어 이용 고객, 노약자, 외국인
- 기능: "서울역 엘리베이터 위치", "용산역에서 KTX 타려면 어디로?", "장애인 도우미 호출 절차"
- 호출 MCP: M2, M7
- ⚠️ **주의:** 장애인 편의시설 데이터 일부는 2018년 이후 미갱신으로 제외됨 → 별도 갱신 요청 필요 (의사결정 사항 #2 참고)

**C3. 고객 응대 보조 Agent**
- 사용자: 고객센터 상담원
- 기능: VOC 패턴 분석, 사전정보공표 자동 검색, FAQ 응답 보조
- 호출 MCP: M10, M7, M1, M2

#### 🛠 직원용 Agent

**E1. 통계·분석 Agent (기획·전략 부서)**
- 사용자: 본사·지역본부 기획·전략·마케팅 담당자
- 기능: "최근 3년 수도권 전철 승하차 변화", "KTX 노선별 수익성", "혼잡도 상위 역"
- 호출 MCP: M3, M4, M5, M6, M7

**E2. 현장운영 Agent (현업·기술 부서)**
- 사용자: 기관사, 차량·시설·전기 부서, 화물 사업소
- 기능: "전동차 X-3500형 제원", "화물 컨테이너 규격 기준", "광운 - 광운대 역간거리"
- 호출 MCP: M4, M5, M6, M7

**E3. 내부지원 Agent (지원·총무·조달 부서)**
- 사용자: 직원 일반 + 총무·조달 담당
- 기능: 직원숙사 예약 안내, 조달 공고 검색, 부서·연락처 조회, 사회공헌 활동 매칭
- 호출 MCP: M8, M9, M7

### 비전공자를 위한 설명
- **데이터 레이어**: 흩어진 파일들을 한곳에 모은 "데이터 창고". 1주~1개월 단위로 공공데이터포털에서 자동으로 끌어와 최신 상태를 유지.
- **도메인 레이어 (MCP)**: 데이터 창고를 주제별로 잘게 쪼개서 AI가 호출하기 쉽게 만든 "창구". 한 창구가 한 도메인을 담당.
- **인터페이스 레이어 (Agent)**: 사용자가 자연어로 대화하는 챗봇. 질문에 따라 필요한 창구만 골라 호출.

## 8. 단계별 추진 로드맵

### Phase 1 — PoC (3개월)
- **목표:** AI Agent 1종 + MCP 서버 1개로 개념 검증
- **선택 권장:** `C1 여행·예매 Agent` + `M1 운행·예약 MCP`  
  → 데이터 시의성·완성도가 가장 높고(차세대예약발매 코드·역코드 등 2024-09 갱신), 사용자 효용도 명확함
- **결과물:** 데모 챗봇, 데이터 카탈로그, 운영 매뉴얼
- **인력:** 데이터 엔지니어 1명, AI 개발자 1명, 도메인 전문가(현업) 1명, PM 1명 (이 기획서 작성자가 겸임 가능)

### Phase 2 — 확장 (3~6개월)
- **목표:** Agent 3종 + MCP 서버 5개로 확장
- **추가 대상:** E1 통계분석 Agent, E2 현장운영 Agent + M3, M4, M6, M7 MCP
- **거버넌스:** 데이터 품질 모니터링 자동화(미갱신 알람, 사문화 데이터 자동 플래깅)

### Phase 3 — 본격 운영 (6~12개월)
- **목표:** Agent 6종 + MCP 서버 10개 전체 가동
- **외부 연동:** 국토부·기상청 OpenAPI 보조 활용, 사내 시스템(KROIS 등)과 단방향 연동 검토
- **확장:** 코레일톡·코레일넷 등 기존 채널에 Agent 임베드

## 9. 기대효과

### 9.1 유형 효과 (정량)

| 항목 | 산출 근거 |
|---|---|
| 직원의 데이터 검색 시간 단축 | 현 평균 X분 → Agent 도입 후 1분 이내 (가설, PoC에서 측정) |

- 데이터 거버넌스 정립: "방치 데이터" 60%를 자동 식별·갱신 요청하는 체계 확보

- 사내 AI 역량 내재화 (외주 의존도 감소)

## 10. 주요 리스크와 대응

| 리스크 | 발생 가능성 | 영향 | 대응 |
|---|:-:|:-:|---|
| 공공데이터 갱신 자체가 부재해 Agent가 잘못된 답변 생성 | 高 | 高 | 사내 데이터 관리 부서와 분기별 갱신 협업 협약 + 답변 시 데이터 기준일 명시 |
| 파일 형식 비표준화로 ETL 실패율 증가 | 中 | 中 | PoC 단계에서 자동 파싱 규칙 사전 검증 |
| 보안·개인정보 이슈 (고객 정보 포함 데이터 식별 누락) | 中 | 高 | 데이터 도메인 분류 단계에서 보안 검토 체크리스트 필수 통과 |
| 직원 사용 부진 (UX 부족) | 高 | 中 | PoC 단계 사용자 인터뷰 → 페르소나별 진입점·말투 차별화 |
